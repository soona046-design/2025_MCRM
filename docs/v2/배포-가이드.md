## 1. 사전 결정 사항 (실 지출 — 착수 전 확인)

- [ ] **VPS 계약**: 2vCPU / 4GB / 서울 리전 (Vultr·DigitalOcean·iwinv 등, 월 2~4만원)
- [ ] **도메인**: 프론트 `<domain>` (Vercel), 백엔드 `api.<domain>` (VPS A레코드)
- [ ] **GitHub 리모트**: `mcrm-v2-backend`는 아직 리모트 없음 → private 리포 생성 후 push

## 2. VPS 최초 셋업 (1회)

```bash
# ① Docker 설치 (Ubuntu 24.04 기준)
curl -fsSL https://get.docker.com | sh

# ② 코드 배치
sudo mkdir -p /srv && cd /srv
git clone git@github.com:<org>/mcrm-v2-backend.git
cd mcrm-v2-backend

# ③ 환경변수
cp .env.production.example .env
php artisan key:generate --show   # 로컬에서 생성해 APP_KEY에 붙여넣기 가능
vi .env                           # DB 비밀번호·CORS·광고 API 키 채우기

# ④ 기동 (entrypoint가 DB 대기 → migrate --force → config:cache 자동 수행)
docker compose -f docker-compose.prod.yml up -d --build
docker compose -f docker-compose.prod.yml logs -f app   # 마이그레이션 확인

# ⑤ 데모 시드는 실행하지 말 것 (db:seed 금지) — 실데이터 적재는 아래 §2.5
```

### 2.5 실데이터 초기 적재 (운영 DB 1회)

원본 CSV(`data/original/`)는 개인정보 포함이라 **git 밖** — 로컬에서 scp로 올린 뒤 로컬과 같은 순서로 적재한다:

```bash
# ① 로컬에서 — 원본 CSV 업로드 (신규문의·광고성과·매출 정리본)
scp -r data/original <user>@<vps>:/srv/mcrm-data/

# ② VPS에서 — 문의+광고 초기 적재 (관리자·직원 계정도 이때 생성됨)
docker compose -f docker-compose.prod.yml exec app \
  php artisan import:real --fresh --path=/srv/mcrm-data/original
# (compose에 /srv/mcrm-data 볼륨 마운트 필요 — app 서비스 volumes에 한 줄 추가)

# ③ 매출 적재 (매출-*-정리.csv는 로컬 파서 산출물을 그대로 사용)
docker compose -f docker-compose.prod.yml exec app \
  php artisan import:revenue --csv=/srv/mcrm-data/original/매출-2025-정리.csv
docker compose -f docker-compose.prod.yml exec app \
  php artisan import:revenue   # 기본 glob을 쓰려면 경로 조정 또는 --csv 반복

# ④ 검증 — 로컬 수치와 대조
curl -s localhost:8100/api/dashboard | python3 -m json.tool | head
```

- 매출 PDF **파서(파이썬)는 로컬 전용** — 서버엔 정리 CSV만 올린다 (갱신 절차: [데이터-갱신-가이드.md](데이터-갱신-가이드.md) §E).
- `import:real --fresh`가 사용자 계정(팀장·직원·관리자·대표)을 함께 생성하므로 별도 관리자 시더가 필요 없다. 비밀번호는 A1 로그인 도입 시 일괄 재설정.

### TLS (api.<domain>)
compose의 nginx는 호스트 8100에 HTTP만 노출한다. 호스트에 Caddy를 얹는 것이 가장 간단:

```bash
sudo apt install caddy
# /etc/caddy/Caddyfile
#   api.<domain> {
#       reverse_proxy localhost:8100
#   }
sudo systemctl reload caddy   # 인증서 자동 발급·갱신
```

## 3. 프론트 (Vercel)

1. Vercel에 `mcrm-v2` 리포 연결 (이미 자체 리포 분리돼 있음)
2. Environment Variables: `NEXT_PUBLIC_API_URL=https://api.<domain>/api`
3. 배포 후 백엔드 `.env`의 `CORS_ALLOWED_ORIGINS`에 Vercel 도메인 등록 → `docker compose ... up -d app` 재기동

## 4. 배포 자동화 (리모트 연결 후)

`mcrm-v2-backend` GitHub 리포 Settings → Secrets에 `VPS_HOST` / `VPS_USER` / `VPS_SSH_KEY` 등록.
이후 main 푸시마다 `.github/workflows/deploy.yml`이 SSH로 `git pull + compose up -d --build` 실행.

## 5. 배포 후 점검 체크리스트

- [ ] `curl https://api.<domain>/api/reports` → 200
- [ ] 프론트 대시보드에서 `/api/dashboard` 렌더 확인 (CORS 오류 없음)
- [ ] `docker compose ps` — app·nginx·mysql·redis·queue·scheduler 6개 모두 healthy/running
- [ ] scheduler 로그에 06:00 KST 수집 예약 확인: `docker compose logs scheduler`
- [ ] 업로드(C4) → 큐 처리 확인: `docker compose logs queue`
- [ ] 보고서 생성(E2) → `POST /api/reports` 201
- [ ] MySQL 백업 크론 등록 (예: 매일 04:00 `mysqldump` → 오브젝트 스토리지)

## 6. 운영 메모

- **로그**: 모든 서비스 stderr → `docker logs`. 장기 보존 필요 시 loki/vector는 후속 검토.
- **큐 재시도**: `queue:work --tries=3 --backoff=300,900,1800` — 설계 §5의 수집 백오프(5/15/30분)와 동일.
- **마이그레이션 경합 방지**: `RUN_MIGRATIONS=1`은 app 서비스에만 설정돼 있음 — queue/scheduler는 마이그레이션을 실행하지 않는다.
- **시드 주의**: `db:seed`는 데모 데이터를 만든다. 운영 DB에는 초기 관리자 계정 시더만 분리 실행(후속 작업).
- **Cafe24 관련**: v1(`mcrm-backend`)의 Cafe24 제약(웹 마이그레이션 우회 등)은 v2에 해당 없음 — v2는 전적으로 VPS 기준.
