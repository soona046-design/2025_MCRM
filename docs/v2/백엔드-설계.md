# M-CRM v2 — 백엔드 설계

> 근거: [기능명세서.md](기능명세서.md) · [IA-화면목록.md](IA-화면목록.md) · 프론트 목스토어(`mcrm-v2/src/lib/mock-*.ts`) — 2026-07-11 manyfast PRD 기준 갱신
> 스택: **Laravel 12 (신규 프로젝트) + MySQL 8 + Redis** — v1(`mcrm-backend`)에서 인증·감사로그·마스킹 모듈 이식

## 1. 아키텍처 개요

```
[네이버 검색광고 API] ─┐
[Meta Marketing API] ──┤→ 수집 파이프라인(정기 06:00 KST + 수동 재수집 + 재시도 큐)
[Google Ads API] ──────┘        ↓
                          ad_metrics_daily ──┐
[수기 입력/업로드] → customers(유입) ──┐    ├→ 집계 엔진(roas_monthly) → 대시보드/보고서 API
[매출 관리 직접 입력] → contracts(신규+기존) ┤    ↓
[외부 매출 웹훅/폴링] → contracts ─────────┘  알림(수집 실패·입력 지연·목표 미달·토큰 만료)
```

**설계 원칙** (명세의 데이터 정책을 그대로 코드 계약으로):

1. **증거 기반 귀속** — `attribution_evidences`에 강한 증거(UTM/클릭ID/전환이벤트ID/캠페인ID)가 1개 이상 있어야 `confirmed`. 없으면 `unknown`으로 저장하고 광고비와 절대 매칭하지 않음 (§1.3.4). 강한 증거는 수기 입력의 캠페인 드롭다운 선택, URL 붙여넣기 자동 파싱, 업로드 UTM 컬럼 자동감지 세 경로로 확보 가능 (S-SSTEUI, S-SDVHBW, S-KCNHUO).
2. **0으로 채우지 않음** — 결측은 NULL. 계약 금액 미입력(NULL)은 ROAS/ROI 분자에서 제외하고 `missing_amount_count`로 별도 집계 (§1.3.5).
3. **월간 = 집계 기준 단위** — 매출 인식 시점은 `contracts.contracted_at`(계약 완료 상태 변경 시각, KST). 월 귀속은 1일 00:00 ~ 말일 23:59 KST 고정. 소급 수정 시 해당 월 재계산 + `recalculation_logs` 기록 (§1.3.5).
4. **이력은 불변** — 단계/필드 변경, 목표 수정, 내보내기, 권한 변경, 토큰 접근 로그 테이블은 INSERT만 허용(모델 레벨 + DB 권한). 관리자 포함 UPDATE/DELETE 불가 (§2.1.1, §2.1.4, S-TDMIEH).
5. **부분 실패 투명성** — 합산 API 응답에 항상 `basis: {succeeded_channels, total_channels, completeness}`를 동봉. 실패 채널 비율 ≥ 임계치(기본 20%, 설정 가능)면 합산값 대신 `data_incomplete` 상태 반환 (§1.1.1).
6. **신규 유입 고객 vs 기존 고객 분리 (신설)** — ROAS 분자는 **유입 관리(customers)와 연결된 신규 유입 고객의 계약 금액만** 사용한다. 재계약·직거래 등 유입 데이터가 없는 "기존 고객" 매출은 ROAS 계산에서 제외하고 ROI(전체 매출 기준) 산출에만 포함한다. 두 지표는 반드시 분자 차이를 라벨로 구분해 응답한다 (S-BOZSZP, S-SVKWCD, F-URURXH).
7. **매출 입력 단일 경로** — 계약 금액 입력 API는 매출 관리 엔드포인트(`/api/contracts`)에서만 제공한다. 유입 관리(customers) 생성/수정 API에는 금액 필드를 두지 않는다 (F-YZXKHM, F-RMMVGI).
8. **수집 모드는 정기/수동 2종뿐** — "매일 자동 갱신", "실시간" 등의 표현은 API·UI 어디에도 쓰지 않는다. 정기 수집(매일 06:00, 전일 데이터, 3채널 동시)과 수동 재수집(하루 최대 3회, 최근 7일 재수집, 토큰 유효 시에만) 두 모드로 고정한다 (F-JOCPFK).

## 2. 배포 환경 — Cafe24 부적합 판정과 대안

### 왜 Cafe24로 안 되는가 (v2 요구사항 대조)

| v2 요구 | Cafe24 공유호스팅 |
|---|---|
| 06:00 정기 수집 + 수동 재수집 + 재시도 백오프(5/15/30분, 상시 스케줄러) | ❌ 웹 cron 수준, 장시간 프로세스 불가 |
| 큐 워커 (업로드 커밋·보고서 생성·소급 재계산) | ❌ Redis 없음, 데몬 불가 |
| PDF 생성 (headless chromium) | ❌ 바이너리 설치 불가 |
| 마이그레이션/배포 자동화 | ❌ SSH 제약 (v1 buglog #8 — 웹 스크립트 우회 중) |

### 권장안: **프론트 Vercel + 백엔드 VPS(Docker)**

- **프론트(mcrm-v2)**: Vercel 무료 티어. Next.js 배포 자동화, 자체 저장소 이미 분리돼 있어 연결만 하면 됨.
- **백엔드**: 국내외 VPS 1대 (예: Vultr/DigitalOcean Seoul 리전 또는 iwinv — 2vCPU/4GB, 월 2~4만원). `docker compose`로 `app(php-fpm) + nginx + mysql + redis + queue-worker + scheduler` 구성. 배포는 GitHub Actions → SSH.
- **Cafe24**: v1 운영 유지용으로만. v2 이관 완료 후 정리.

> 대안 B(PaaS: Railway/Render): 관리 부담 최소지만 비용 예측이 어렵고 국내 리전 부재. 트래픽이 작은 사내 도구라 VPS가 비용·제어 면에서 유리. **VPS 계약은 실 지출이므로 착수 전 확인 필요.**

## 3. DB 스키마

UUID PK, `timestamps` 기본. `(불변)` = INSERT 전용.

### 3.1 사용자·권한 (§2.1.4~5)

```
users            id, name, email(unique), password, role(enum: representative|director|manager|staff|admin),
                 is_active, last_login_at
user_permissions user_id, permission_key, allowed(bool)          -- 아이디별 개별 오버라이드(상한: 역할 기본권한)
permission_logs  (불변) actor_id, target_user_id, changes(json)  -- 권한 변경 이력
```

- 역할 매핑: 대표=representative, 이사=director, 팀장=manager, 담당자=staff, 관리자=admin.
- 연락처 마스킹은 **응답 직전** 역할 기준 적용(대표/이사/팀장 마스킹, 담당자/관리자 원본 — §2.1.4). 원본은 암호화 컬럼에만.

### 3.2 광고 채널·수집 (§1.1)

```
ad_channels        code(pk: naver|meta|google), name, credentials(encrypted json, AES-256),
                   status(connected|disconnected), connected_at, last_collected_at, token_expires_at
token_access_logs  (불변, S-TDMIEH) id, channel_code, event(issued|refreshed|manual_reconnect|revoked|refresh_failed),
                   actor_id(null=시스템), occurred_at
ad_metrics_daily   id, channel_code, campaign_id, campaign_name, keyword(null),
                   date(KST 귀속일), impressions, clicks, cost
                   UNIQUE(channel_code, campaign_id, keyword, date)  -- 소급 재수집 시 upsert
collection_runs    (불변, 90일 보존) id, channel_code, trigger_type(scheduled|manual), started_at, finished_at,
                   status(running|success|delayed|failed), error, retry_count
```

- 수집 상태 판정(§1.1.2): 마지막 성공 24h 이내=`완료`, 06:00 초과=`지연`, API 오류=`실패`. 판정은 저장하지 않고 `collection_runs`에서 파생.
- 수집 모드(F-JOCPFK): `scheduled`=매일 06:00 KST 전일 데이터 3채널 동시, `manual`=팀장/관리자가 트리거, 최근 7일 재수집, 하루 최대 3회·토큰 유효 시에만.
- 재시도: API 실패(타임아웃/5xx) 시 최대 3회(5분→15분→30분 백오프), 토큰 만료는 재시도 없이 즉시 알림.
- 알림 단계(§1.1.1): 실패 1h→인앱(관리자), 6h→이메일, 24h→전체 배너 + 이메일. 토큰 만료 7일 전 사전 알림(팀장+관리자).
- 토큰 원문은 어떤 역할도 직접 열람 불가. 읽기/재발급은 서버사이드 프로세스만 수행(S-TDMIEH).

### 3.3 고객·퍼널 (§2)

```
customers        id, name, phone_encrypted, phone_masked, consult_channel(enum: phone|kakao|web_form|other),
                 source_category(enum: online_ad|online_organic|offline_ad|referral|other),
                 ad_channel_code(null), campaign_id(null), campaign_name(null),
                 attribution(enum: confirmed|unknown|na),   -- 증거 판정 결과 캐시
                 stage(enum: inflow_event|lead|consult|proposal|contract),  -- 5단계(변경됨, 구 4단계 대체)
                 owner_id, inflow_date, memo, upload_batch_id(null), anonymized_at(null)
attribution_evidences  customer_id, type(utm_source|utm_medium|utm_campaign|click_id|conversion_event_id|campaign_id), value,
                 captured_via(dropdown|url_paste|upload_autodetect|manual)  -- 확보 경로(신설)
change_logs      (불변) customer_id, field, old_value, new_value, actor_id, actor_role, reason(null)
                 -- 단계 변경·담당자 변경 전부 여기에 (§2.1.1 프론트 이력 타임라인과 1:1). 계약 금액 변경은 contracts 쪽 이력 사용
inflow_events    id, occurred_at, consult_channel, contact_masked, owner_id,
                 matched_customer_id(null), source(manual|upload|form|call_log)
                 -- 입력률 KPI 분모 (§2.1: 24h 내 matched 비율 ≥ 90%). 미매칭 = 미입력/지연 목록
```

- `customers` 생성/수정 API에는 계약 금액 필드를 두지 않는다. `stage=contract` 진입 시 프론트에서 "매출 관리에서 계약 금액을 등록하세요" 안내만 표시(원칙 7).
- 캠페인 드롭다운으로 선택된 건은 `attribution_evidences.captured_via=dropdown` + `campaign_id`로 강한 증거 자동 확보(S-SSTEUI). UTM 붙여넣기 파싱 결과는 `url_paste`(S-SDVHBW).

### 3.4 업로드 (§2.2)

```
upload_batches    id, filename, uploader_id, mapping(json), status(validating|committed|failed),
                  total_rows, confirmed_rows, ok_rows, demoted_rows, error_rows
upload_row_errors batch_id, row_no, message    -- 행 단위 오류 (프론트 3단계 마법사와 동일 계약)
```

- 검증·강등 규칙은 프론트 `upload/page.tsx`의 `validateRow`와 동일 로직을 서버가 최종 판정(관대 모드 기본, §2.2.4).
- UTM 컬럼 자동감지(S-KCNHUO): `mapping.json`에 감지된 컬럼을 `utm_source|utm_medium|utm_campaign|click_id|campaign_id` 그룹으로 분리 제안. 저장된 매핑 규칙은 `uploader_id` 단위로 재사용 제안.
- 처리 SLA(F-HFHQRW): 100행 이하 30초, 100~1,000행 3분, 1,000행 초과 10분. 비동기 큐 처리, 전체 실패 시 2회 재시도.

### 3.5 매출·목표 (§7, §1.3) — 신규 유입/기존 고객 구분 반영

```
contracts        id, customer_type(enum: new_lead|existing),
                 lead_customer_id(null, FK customers — new_lead일 때만),
                 existing_customer_id(null, FK existing_customers — existing일 때만),
                 amount(null=미입력), status(enum: ongoing|done|canceled),
                 contracted_at(KST — 매출 인식 시점), external_source(manual|webhook|api|file),
                 external_amount(null)   -- 외부 연동값 병기 (§1.3.7: 연동 우선, 수기는 참고)
                 -- CHECK: customer_type=new_lead ⇒ lead_customer_id NOT NULL, existing_customer_id NULL (역also)
existing_customers  id, name, phone_encrypted, phone_masked, owner_id, first_contract_at,
                 memo, merged_from_customer_id(null)  -- 유입고객에서 수동 전환된 경우 원본 참조(F-NTYUDC)
revenue_goals    month(YYYY-MM), channel_code(null=전체), customer_type(null=전체|new_lead|existing), amount
                 UNIQUE(month, channel_code, customer_type)
goal_logs        (불변) month, from_amount(null), to_amount, actor_id
roas_monthly     (집계 캐시) month, channel_code(null=전체), campaign_id(null),
                 ad_cost, revenue_confirmed(신규 유입 고객 계약금액 합산 — ROAS 분자),
                 total_revenue(신규+기존 전체 계약금액 합산 — ROI 분자),
                 contracts_count, missing_amount_count, input_rate, completeness,
                 roas(null 가능), roi(null 가능)  -- 분모/분자 결측 시 둘 다 null
recalc_logs      (불변) scope(json), reason(retroactive_amount|recollection|goal_change), triggered_by
```

- ROAS 산출 가능/불가 매트릭스(§1.3.2)를 그대로: 광고비∅→`광고비 대기`, 금액∅→`금액 정보 없음`, 확정 전환 0건→`계산 불가`(0배 반환 금지 — API가 `roas: null` + `reason` 반환). ROI도 동일 원칙(`roi: null` + `reason`).
- 매출 직접 입력(`POST /api/contracts`)은 `customer_type`에 따라 `lead_customer_id`(유입 고객 검색·선택) 또는 `existing_customer_id`(기존 고객 검색·선택 또는 신규 등록)를 요구. 유입 관리에 연결되지 않은 고객으로 입력하면 서버가 `existing_customers`에 자동 등록(F-NTYUDC).
- 동일 이름/연락처가 `existing_customers`에 이미 있으면 병합 확인 응답(`409 possible_duplicate` + 후보 목록)을 반환하고, 클라이언트가 병합/신규등록을 선택.

### 3.6 보고서 (§3)

```
reports         id, type(weekly|monthly|yearly|custom), period_start, period_end, filters(json),
                version, data_snapshot(json), status(generating|ready|failed),
                data_updated_at(null — 소급 반영 배지), created_by,
                handed_over_from(null — 대시보드 필터 인계 여부, S-LMUKBI)
                -- 동일 (type, period, filters 해시) 재생성 시 version+1 (§3.3.3), 12개월/100버전 보존
report_exports  (불변, 90일) report_id, format(pdf|excel), actor_id, actor_role, filename, filters_summary
saved_filters   user_id, name(≤30자), filters(json)   -- 사용자당 최대 10개 (§3.2.1)
```

### 3.7 알림·설정·개인정보 (§1.1.1, §2.1.6)

```
notifications    user_id(null=전체), type(collection_failed|input_overdue|amount_missing|goal_below|
                 data_updated|token_expiring|token_expired|...), payload(json), read_at
system_settings  key, value(json)   -- 실패 임계 20%·입력률 임계 70%·목표 알림 70%·매칭키 엄격/경고 모드·보관 기간 등 (관리자 조정)
privacy_logs     (불변) customer_id, action(anonymize|delete), trigger(expiry|request|termination), actor_id
```

- 익명화 배치(일 1회): 보관 기간(기본 12개월) 만료 리드의 `name/phone_encrypted` 해시 대체, `anonymized_at` 기록. 집계 테이블은 비식별이므로 유지 (§2.1.6). `existing_customers`도 동일 정책 적용 대상.
- `system_settings`의 매칭 키 모드(`attribution_mode: strict|lenient`, 기본 lenient)는 수기 입력 시 강한 증거 없는 온라인 광고 건을 저장 차단할지(strict) 경고 후 허용할지(lenient) 결정(S-RBUWWG).

## 4. API 계약 (프론트 목스토어 → 실제 엔드포인트)

인증: Sanctum 토큰. 공통 규약 — 합산 응답에 `basis`(수집 성공 채널 N/전체 N, completeness), 결측은 `null`+`reason`, 목록은 `?page=&per_page=20`.

| 프론트 (현재 목) | 엔드포인트 | 비고 |
|---|---|---|
| 대시보드 KPI·차트 | `GET /api/dashboard?period=month&month=2026-07` | KPI 4종(ROAS·광고비·전체매출·전환수)+ROAS·ROI 이중축 추이+퍼널(5단계)+비중 일괄 |
| 신뢰도 바 (`DataTrustContext`) | `GET /api/collection/status` · `POST /api/collection/{channel}/retry` | 상단 바 폴링(60s), retry = 수동 재수집(하루 3회 제한) |
| `customers` 목록 (C1 URL 파라미터) | `GET /api/customers?q=&stage=&source=&owner=&sort=&page=` | 쿼리 키를 프론트와 동일하게. `stage` 값은 5단계 enum |
| 고객 상세/이력 (C2) | `GET /api/customers/{id}` · `GET …/{id}/logs` | 이력 = change_logs |
| 단계 변경 (역행 확인) | `POST /api/customers/{id}/stage {to, reason?}` | 서버도 역행 검증·이력 기록 |
| 수기 입력 (C3) | `POST /api/customers` | 계약 금액 필드 없음. 증거 필드(드롭다운 campaign_id 또는 UTM/클릭ID) → attribution 판정 |

## 5. 수집 파이프라인·배치

| 잡 | 트리거 | 내용 |
|---|---|---|
| `CollectAdMetrics{Naver,Meta,Google}` | 스케줄러 06:00 KST (정기 수집, `trigger_type=scheduled`) | 전일 데이터 upsert, 3채널 동시 실행. 완료 후 10분 이내 대시보드 반영. 실패 시 백오프 재시도(5분→15분→30분, 최대 3회) → collection_runs 기록 → 알림 단계 적용 |
| `ManualRecollect{channel}` | 팀장/관리자 "지금 수집" 클릭 (`trigger_type=manual`) | 최근 7일 데이터 재수집, 완료 후 5분 이내 반영. 하루 최대 3회, 토큰 유효 시에만 실행 |
| `RecalculateMonth` | 금액 소급 수정·재수집 성공 시 큐 | roas_monthly의 revenue_confirmed(신규유입)/total_revenue(전체) 각각 재계산 + recalc_logs + 관련 보고서 `data_updated_at` 갱신 |
| `CommitUploadBatch` | 업로드 커밋 | 검증→customers/attribution_evidences 생성, 행 오류 기록. UTM 컬럼 자동감지 매핑 재사용 |
| `MergeExistingCustomer` | 매출 직접 입력 시 중복 후보 발견 | 동일 연락처 existing_customers 병합/신규 등록 처리 |
| `GenerateReport` | 보고서 생성 요청 | 스냅샷 집계 → PDF(Browsershot/chromium)·Excel(OpenSpout) |
| `CheckOverdueInputs` | 매시 | 24h/48h 초과 판정 → 담당자/팀장 알림 (§1.3.5 §6) |
| `CheckTokenExpiry` | 매일 | 만료 7일 전 채널 토큰 → 팀장+관리자 이메일 알림 (S-JVSEJT) |
| `AnonymizeExpired` | 일 1회 03:00 | 보관 만료 익명화(customers + existing_customers) + privacy_logs |

**광고 API 리드타임 주의** — 코드보다 심사가 오래 걸립니다: 네이버 검색광고 API(라이선스 키 발급), Meta Marketing API(앱 검수, Ads Read 권한), Google Ads API(developer token 심사, 수 주 소요 가능). **B3 시작 전에 미리 신청**해둘 것.

## 7. 구현 로드맵

| 단계 | 범위 | 프론트 연결 |
|---|---|---|
| **B0** | Laravel 12 신규 + docker compose + CI/CD + 마이그레이션 골격 | — (배포 환경 확정 필요) |
| **B1** | 인증·권한·사용자 + 고객/퍼널 CRUD(5단계 stage) + 업로드(UTM 자동감지) + 미입력 현황 | M2 화면 실데이터 전환 |
