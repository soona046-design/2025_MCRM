## 1. 핵심 변경 요약

| 변경 | 이전 | 이후 |
|---|---|---|
| 퍼널 단계 | 문의 → 상담 → 예약 → 계약 (4단계) | 유입 이벤트 → 리드 → 상담 → 견적/제안 → 계약 (5단계) |
| 계약 금액 입력 위치 | 고객 상세 화면(유입 관리) | **매출 관리 전용** — 유입 관리에서는 입력 UI 자체가 없음 |
| 매출 지표 | ROAS 단일 (계약 금액 전체 ÷ 광고비) | **ROAS**(신규 유입 고객 매출만) / **ROI**(신규+기존 전체 매출) 분리, 라벨 상시 병기 |
| 고객 구분 | 없음 | **신규 유입 고객**(유입 관리 연결) vs **기존 고객**(재계약·직거래) |
| 광고 매칭 증거 입력 | UTM/클릭ID 수기 입력만 | + 캠페인 드롭다운 자동연동, URL 붙여넣기 자동파싱 |
| 수집 모드 표현 | (미정) | 정기 수집(06:00) / **수동 재수집**("지금 수집" 버튼) 2모드 |
| 화면 수 | 20개 | **21개** — 기존 고객 목록/상세(D5) 신규 |

## 3. 수정 파일

### 데이터 레이어
- **`src/lib/mock-customers.ts`** — `Stage` 타입을 5단계로 교체, `Customer`/`NewCustomerInput`에서 `contractAmount` 제거, 광고 채널·캠페인 드롭다운 데이터(`AD_CHANNELS`, `CAMPAIGNS_BY_CHANNEL`) 추가
- **`src/lib/mock-revenue.ts`** — 전면 재작성. `contracts`가 계약 금액을 단독 소유(고객 스토어에서 파생하지 않음), `customerType`(신규 유입/기존) 필드, `addContract`/`updateContract`(이력 기록), `roasRoiTrend()`(이중축 추이), `GOAL_SCOPES`(고객유형별 목표)

### 고객 유입 관리
- **`customers/[id]/page.tsx`** — 계약 금액 카드·다이얼로그 제거 → "매출 관리에서 계약 금액 등록" 링크로 대체. 단계 역행 시 사유 선택(재검토 필요/오류 수정/기타) 필수화
- **`customers/new/page.tsx`** — 계약 금액 필드 제거. 광고 채널 선택 시 캠페인 드롭다운 자동 연동(목록에 없으면 직접 입력 전환), "URL 붙여넣기" 입력란으로 UTM/클릭ID 자동 파싱
- **`customers/upload/page.tsx`** — `STAGE_ALLOWED` 5단계로, `contractAmount` 컬럼·검증·템플릿/샘플 CSV에서 전부 제거

### 매출 관리
- **`revenue/page.tsx`(D1)** — 고객유형 탭(전체/신규 유입/기존), "매출 직접 입력" 다이얼로그(신규=유입 고객 검색연결, 기존=자동완성 검색 또는 신규 등록+중복 병합 확인), 금액 클릭 시 인라인 편집, 목록에 고객유형 컬럼 추가
- **`revenue/analytics/page.tsx`(D4)** — 고객유형 필터 탭, ROAS(좌축)·ROI(우축) 이중축 차트 추가, 채널 기여도는 신규 유입 전용(기존 선택 시 안내 문구)
- **`revenue/goals/page.tsx`(D3)** — 목표를 전체/신규 유입/기존 탭으로 세분화, 이력에 구분 배지 표시
- **`components/revenue/revenue-tabs.tsx`** — "기존 고객 목록" 탭 추가

### 대시보드(B1)
- **`components/dashboard/kpi-cards.tsx`** — "계약 금액" 카드 → "전체 매출"(전체 포함 라벨)로 교체, ROAS 카드에 "신규 유입 기준" 라벨 추가
- **`components/dashboard/roas-trend.tsx`** — 채널별 ROAS 추이 → **ROAS·ROI 이중축 추이**로 전면 교체(`roasRoiTrend()` 사용, 실제 계산값 반영)
- **`components/dashboard/funnel-chart.tsx`** — 5단계(유입 이벤트→리드→상담→견적/제안→계약)로 갱신
- **`app/(main)/page.tsx`** — 레이아웃을 명세 순서(요약카드 → ROAS·ROI 추이 → 채널비교+퍼널 → 채널비중 → 신뢰도)로 재배치, "지금 수집"(수동 재수집) 버튼 추가

### 내비게이션
- **`components/layout/app-shell.tsx`** — 사이드바 "매출 관리" 하위에 "기존 고객 목록" 추가, 상세 페이지 활성 상태 매칭 규칙 보강

## 4. 반영하지 않은 항목 (범위 밖)

- **백엔드**: v2 Laravel 신규 프로젝트는 아직 미착수(VPS 등 인프라 결정 필요, 별도 확인 후 진행)
- **B2/B3(채널·캠페인 상세 분석), E1~E4(보고서), F1~F5(설정)**: 기존에도 목데이터 스캐폴드가 없던 화면이라 이번 범위에서 신규 구축하지 않음 — 사이드바에 "준비 중"으로 표시된 채 유지
- **브라우저 인터랙션 테스트**: 이 세션에서 Claude in Chrome 확장이 연결되지 않아 폼 제출·다이얼로그 클릭 등 실제 조작 확인은 못 했음. curl 기반 SSR 콘텐츠 검증(라우트 200, 신규 문구 노출)까지만 완료

## 5. 확인 방법

```bash
cd mcrm-v2
npm run dev   # http://localhost:3000
```

주요 확인 포인트:
- `/customers/new` — 광고 채널 선택 → 캠페인 드롭다운, URL 붙여넣기 자동파싱

- `/revenue/existing`, `/revenue/existing/[id]` — 신규 화면

## 6. 2026-07-12 폴리싱 — 경미 관찰 2건 조치

### ① 기존 고객 상세 "재계약 주기 안내" 일부 미노출
- **원인**: `revenue/existing/[id]/page.tsx`가 "오늘" 기준으로 `new Date()`(실제 시스템 시각)를 썼는데, 목데이터의 계약일은 전부 2026년으로 고정돼 있어 실제 시계와 어긋나면 경과일 계산이 항상 0 또는 비정상적으로 커짐 — 다른 화면(`mock-customers.ts`의 `NOW_REF`, `customers/page.tsx`의 `NOW`)은 전부 앱 내부 고정 기준시각을 쓰는 것과 불일치했던 게 근본 원인.
- **조치**: `mock-revenue.ts`가 이미 내보내는 `CURRENT_MONTH`(`2026-07`)·`DAYS_ELAPSED`(`10`)로 고정 기준일(`2026-07-10`)을 구성해 사용하도록 수정. 부수적으로 `daysBetween`의 불필요한 `Math.max(0, …)` 클램프도 제거(미래 날짜 계약과의 차이를 자연스럽게 음수로 표현해 조건이 올바르게 `false`가 되도록).
- **부가 조치**: 이 기준일로 재계산하면 목데이터 생성 규칙상 최근 계약이 항상 "충분히 최근"이라 배너가 사실상 아무에게도 안 뜨는 2차 문제가 드러나, `mock-revenue.ts`의 기존 고객 계약 시드에서 계약 2건 이상 고객의 절반(`ei % 2 === 0`)은 마지막 계약을 의도적으로 3개월 더 이전으로 배치하도록 조정. 계약 1건 고객은 애초에 주기를 계산할 수 없어 항상 미노출인 것이 정상 동작.
- **검증**: `ex1000~ex1011` 12명 중 `ex1002·ex1004·ex1008·ex1010`(계약 2건 이상 + 오래된 마지막 계약)에서만 배너 노출 확인, 나머지는 미노출 — 데이터 조건부 동작임을 확인.

### ② ROAS·ROI 차트 당월 라벨 겹침
- **원인**: 당월(7월) ROAS가 급락(0.45배)하고 ROI가 마이너스(-17.8%)로 나오면서 두 값의 화면상 y좌표가 서로 가까워졌는데, 라벨 위치를 각 포인트 기준 고정 오프셋(`-8`/`+16` 등)으로만 계산해 겹침 방지 로직이 없었음.
- **조치**: `components/dashboard/roas-trend.tsx`와 `revenue/analytics/page.tsx`의 `RoasRoiChart` 두 곳 모두에 최소 간격(14~16px) 보장 로직을 추가 — 두 라벨이 너무 가까우면 아래쪽(ROI) 라벨을 밀어내고, 캔버스 하단을 벗어날 경우엔 위쪽(ROAS) 라벨을 대신 밀어 올리는 방식으로 항상 최소 간격과 캔버스 내부 범위를 보장.
- **검증**: 렌더링된 SVG에서 두 라벨의 실제 `y` 좌표 차이가 대시보드 43px, 분석 화면 50px로 벌어져 있음을 확인(둘 다 `MIN_GAP` 이상, 캔버스 범위 내).
